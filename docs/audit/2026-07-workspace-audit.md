# Statohub Workspace Audit — 2026-07-05

Read-only audit. Every claim below cites a repo-relative path. 99 commits total, first commit `2026-06-13`, HEAD `c1a8e30`.

## Summary

The repo is materially more mature than the strategy review assumed — most `[verify]` items resolve as **already built**, not missing. Astro 4.16 static site, `trailingSlash:'always'`, deployed via GitHub Actions → Cloudflare Pages with a real gate-then-deploy pipeline ([`astro.config.mjs`](../../astro.config.mjs), [`.github/workflows/deploy.yml`](../../.github/workflows/deploy.yml)). Calculators are **fully componentized**: 26 pure engines in `src/calc/**`, one `registry.ts`, one `<StatCalc>` renderer, 89 unit tests. Content schemas are Zod-enforced, not conventional. Internal linking is a typed, build-gated system (`src/lib/links.ts` + `scripts/check-links.mjs`) — the single strongest piece of infrastructure in the repo. Two real defects were found and are worth immediate fixes: a **live, rendering broken glyph** on every article H2 ([`src/styles/global.css:1791`](../../src/styles/global.css#L1791)), and **duplicate/conflicting CSS class definitions** from two design-system generations coexisting in one file. Tertiary text tokens fail WCAG AA contrast in both light and dark mode. No images are used anywhere in content. Calculator tests validate closed-form happy paths but are not a systematic textbook-canonical-value regression harness — the strategy review's top P0 recommendation is partially, not fully, met.

## Confirmed strengths

- Typed route registry + build-time link gate makes "zero internal redirects/404s" a compile/build enforced invariant, not a policy — [`src/lib/links.ts`](../../src/lib/links.ts), [`scripts/check-links.mjs`](../../scripts/check-links.mjs), route ids codegen'd from real content by [`scripts/gen-route-ids.mjs`](../../scripts/gen-route-ids.mjs) so they can't drift.
- Calculator math is fully centralized: 26 engines under `src/calc/*.ts`, each pure (no Astro/DOM imports — grep confirmed clean in prior sessions per [`CLAUDE.md`](../../CLAUDE.md)), one `registry.ts` keyed by engine name, one `<StatCalc>` dual-deploys page/embed variants from the same YAML config ([`src/components/StatCalc.astro`](../../src/components/StatCalc.astro)).
- Content schema is Zod-enforced with cross-collection `reference()` integrity (articles → categories/calculators, calculator-content → calculators) — [`src/content/config.ts`](../../src/content/config.ts). Sampled 3 articles across 2 silos; all conform (single H1 via `article.h1 ?? article.title`, consistent frontmatter, body starts at H2).
- CI is a real gate: `astro check` → `vitest` (89/89 passing) → `astro build` (which itself runs the link gate) → deploy, `concurrency` guarded against overlapping deploys — [`.github/workflows/deploy.yml`](../../.github/workflows/deploy.yml).
- Security headers + baseline CSP are shipped via Cloudflare `_headers` — [`public/_headers`](../../public/_headers).
- No secrets tracked in git; `.env*` correctly gitignored — [`.gitignore`](../../.gitignore).
- Structured data (BreadcrumbList, Article, SoftwareApplication, WebSite/Organization) is generated from pure helpers that route every URL through the same `url()` function the link gate checks — [`src/lib/schema.ts`](../../src/lib/schema.ts).

## Verify-list answers

- **"Framework is a static generator suited to GitHub Pages"** — **Confirmed**, but it is not GitHub Pages: Astro 4.16.18 (`type:'module'`, [`package.json`](../../package.json)) deployed to **Cloudflare Pages** via GitHub Actions ([`.github/workflows/deploy.yml`](../../.github/workflows/deploy.yml), [`wrangler.toml`](../../wrangler.toml)). Functionally equivalent for the strategy review's purposes (static, zero backend), but the specific host differs from the assumption.
- **"Calculators are/aren't componentized enough for an embed program"** — **Confirmed componentized, and unusually well so.** One `<StatCalc>` component ([`src/components/StatCalc.astro`](../../src/components/StatCalc.astro)) dual-deploys `variant="page"` and `variant="embed"` from the same YAML config and the same pure engine in `src/calc/registry.ts`. This is a stronger starting position for an embeddable-calculator PLG program than the review assumed — the "embed" variant already exists and ships today on every article that carries a `calculator:` frontmatter field.
- **"Existing content depth per silo"** — **Partially confirmed, numbers corrected.** Articles per category: descriptive-statistics 10, probability-distributions 4, regression-correlation 3, foundations 2, inferential-statistics 2, combinatorics 0. Calculators per category: descriptive-statistics 13, inferential-statistics 7, probability-distributions 4, combinatorics 2, regression-correlation 2, foundations 1. The review's assumption that "Descriptive Stats and Probability" are the two deepest silos is correct for Descriptive Stats (clear leader) but Probability-distributions is a distant second on articles (4 vs. 10) — Regression-correlation is close behind it. Recomputing the quiz-silo pick against these real numbers is worth 5 minutes before committing to the two-silo experiment.
- **"Presence/absence of any CLAUDE.md, memory files, or automation"** — **Confirmed present, and extensive.** [`CLAUDE.md`](../../CLAUDE.md) (a very long, dated session log — the workflow report's "knowledge lives in chat histories" concern is only half true: a lot of it *is* written down, just append-only and un-indexed), [`AGENTS.md`](../../AGENTS.md) (Codex entry point), [`.claude/agents/`](../../.claude/agents) (2 subagents: `stats-article-writer.md`, `stats-article-reviewer.md`), [`.claude/seo-playbook.md`](../../.claude/seo-playbook.md), [`.claude/skills/write-article/`](../../.claude/skills/write-article). No `hooks` config and no `.mcp.json` found in-repo. This repo already has real automation (scheduled cloud routines for article/calc-prose publishing, referenced throughout `CLAUDE.md`) beyond what the strategy review's "6 agents, memory MVP" section assumed as greenfield.
- **"Mobile calculator ergonomics and dark-mode contrast risks"** — **Contrast risk confirmed** (see findings table: `--ink-3` fails AA in both themes). Mobile ergonomics could not be fully verified statically — [`global.css`](../../src/styles/global.css) has a `@media (max-width: 980px)` breakpoint collapsing `.panel-grid` (the calculator's two-column input/output layout) to one column, which is the right pattern, but actual on-device tap-target sizing and viewport behavior needs a live/emulated check this audit did not run (no browser tooling used — static-file review only).
